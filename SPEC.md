## 1. プロジェクト概要

### 1.1 Pokemon Champions とは

Pokemon Champions は The Pokemon Company が開発した**バトル特化型の公式ゲーム**。Pokemon Stadium シリーズの精神的後継で、RPG 要素はなくバトルのみに焦点を当てている。

- **プラットフォーム**: Nintendo Switch / Switch 2（4月8日）、iOS/Android（2026年夏）
- **基本プレイ無料**（Switch 版は有料版あり）
- **VGC 公式大会タイトル**: 2026年ポケモン世界大会（8月28-30日, サンフランシスコ）から使用
- **クロスプラットフォーム対戦対応**
- **Pokemon HOME 連携**

### 1.2 シミュレーターの目的

Pokemon Champions の対戦環境において、以下を実現するダメージシミュレーターを構築する:

- 正確なダメージ計算（新 SP システム対応）
- メガシンカを含む全ギミック対応
- VGC（ダブルバトル）特化の UI/UX
- バランスパッチに即座に追従可能なアーキテクチャ
- モバイルファースト設計
- 既存ツールの課題を解決する革新的機能

### 1.3 競合状況

| ツール | 状態 | 特徴 | 課題 |
|--------|------|------|------|
| Smogon Calc | Champions 未対応 | 業界標準の精度 | モバイル不可、SP 未対応 |
| Pikalytics | Champions 一部対応 | 使用率データ連携 | 計算機能は限定的 |
| Champions Lab | Champions 対応 | AI バトルシミュ、200万戦 | ダメージ計算の詳細度不明 |
| Pokesample | Champions 対応（開発中） | SP システム対応 | まだ開発途中 |
| VGC Multicalc | Champions 未対応 | 複数同時計算 | Smogon calc 依存 |

**チャンス**: Pokemon Showdown が DMCA リスクにさらされている今、独立したシミュレーターへの需要は過去最高。

## 2. Pokemon Champions 固有の仕様

### 2.1 ステータスシステム（SP システム）

**従来の IV/EV システムは廃止され、Stat Points (SP) に置き換えられた。**

| 項目 | 従来（SV まで） | Champions |
|------|-----------------|-----------|
| 個体値 (IV) | 0-31（ランダム） | **廃止（常に 31 相当）** |
| 努力値 (EV) | 0-252/stat, 合計 510 | **Stat Points: 0-32/stat, 合計 66** |
| 性格 | Nature（25 種） | **Stat Alignment**（同一の 25 種、名前変更） |
| レベル | 1-100 | **常に 50（オートレベル）** |

#### SP ⇔ EV 変換

```
1 SP = 8 EV = Lv50 で 1 ステータスポイント
66 SP = 528 EV 相当
32 SP max = 256 EV 相当
```

#### ステータス計算式（Lv50 固定）

**HP:**
```
HP = floor((2 * Base + 31) * 50 / 100) + 50 + 10 + SP_HP
   = floor(Base + 15.5) + 60 + SP_HP
   = Base + 75 + SP_HP  （Base が偶数の場合）
```

**その他（攻撃、防御、特攻、特防、素早さ）:**
```
Stat = floor((floor((2 * Base + 31) * 50 / 100) + 5 + SP) * NatureModifier)
     = floor((Base + 20 + SP) * NatureModifier)  （Base が偶数の場合、概算）
```

NatureModifier: 上昇 = 1.1 / 無補正 = 1.0 / 下降 = 0.9

#### シミュレーターへの影響

- **IV 入力フィールドは不要**（常に 31）
- **SP 入力**: 各ステータスに 0-32 のスライダー/入力、合計 66 上限
- **「2 つのステータスをフル + 残り 2」が基本配分**（32+32+2 = 66）
- 従来の EV スプレッド（252/252/4 など）の概念は SP に変換して表示

### 2.2 メガシンカ（ランチ時のギミック）

- ポケモンに**メガストーン**を持たせる（持ち物枠を消費）

## 3. ダメージ計算式

### 3.1 コア計算式（Gen 5-9 共通、Champions でも不変）

```
Damage = floor(floor(floor(2 * Level / 5 + 2) * Power * A / D) / 50 + 2)
         * Targets * PB * Weather * GlaiveRush * Critical * random * STAB * Type * Burn * other * ZMove * TeraShield
```

**Champions では Level = 50 固定なので:**

```
基本値 = floor(floor(22 * Power * A / D) / 50 + 2)
```

各乗算後に「0.5 を 0 方向に丸める」端数処理を適用。

### 3.2 全修正値一覧

#### Targets（範囲技補正）

| 条件 | 値 |
|------|-----|
| ダブルで複数対象 | **0.75** |
| 単体対象 or 対象が 1 体のみ | 1.0 |

#### Parental Bond（おやこあい）

| 条件 | 値 |
|------|-----|
| メガガルーラ 2 撃目 | **0.25** |
| それ以外 | 1.0 |

#### Weather（天候）

| 条件 | 値 |
|------|-----|
| 水技 in 雨 | **1.5** |
| 炎技 in 晴れ | **1.5** |
| 水技 in 晴れ | **0.5** |
| 炎技 in 雨 | **0.5** |
| ハイドロスチーム in 晴れ | **1.5** |
| その他 | 1.0 |

天候の副次効果（ステータス上の影響）:
- 砂嵐: 岩タイプの特防 **1.5 倍**（ステータスに直接適用）
- 雪: 氷タイプの防御 **1.5 倍**（ステータスに直接適用）

#### Critical（急所）

| 条件 | 値 |
|------|-----|
| 急所 | **1.5** |
| 通常 | 1.0 |

急所の追加効果:
- 攻撃側の能力低下を無視
- 防御側の能力上昇を無視
- リフレクター・ひかりのかべ・オーロラベールを無視

#### Random（乱数）

```
random = randint(85, 100) / 100
```

16 段階（85, 86, ... 100）の一様分布。

#### STAB（タイプ一致ボーナス）

| 条件 | 値 |
|------|-----|
| タイプ一致 | **1.5** |
| タイプ一致 + てきおうりょく | **2.0** |
| 不一致 | 1.0 |

テラスタル STAB（将来のテラスタル実装時）:

| 条件 | 値 |
|------|-----|
| テラタイプ = 元タイプ、テラタイプの技使用 | **2.0** |
| 上記 + てきおうりょく | **2.25** |
| テラタイプ ≠ 元タイプ、テラタイプの技使用 | **1.5** |
| テラスタル後、元タイプの技使用 | **1.5** |
| ステラテラ: 各タイプ初回 | **2.0** |
| ステラテラ: 同タイプ 2 回目以降 | **1.5** |

#### Type（タイプ相性）

| 結果 | 倍率 |
|------|------|
| 4 倍弱点 | **4.0** |
| 2 倍弱点 | **2.0** |
| 等倍 | 1.0 |
| 半減 | **0.5** |
| 1/4 耐性 | **0.25** |
| 無効 | **0.0** |

#### Burn（やけど）

| 条件 | 値 |
|------|-----|
| やけど状態 + 物理技 + こんじょう以外 | **0.5** |
| やけど + からげんき | 1.0 |
| その他 | 1.0 |

#### Other（複合修正値）

防御壁:

| 効果 | シングル | ダブル |
|------|---------|--------|

### 5.2 差別化機能

#### D. マルチ計算モード

既存ツール最大の弱点を解決:

- **1 vs Team**: 攻撃側 1 体 vs チーム 6 体を一括計算
- **Team vs 1**: チーム 6 体 vs 防御側 1 体を一括計算
- **Team vs Team**: チーム 6 体 vs チーム 6 体のマトリックス表示

#### E. SP 最適化ツール

「このポケモンで A を確定 1 発にしつつ、B の攻撃を耐えたい」→ 最小 SP 配分を自動算出

```
入力:
  - KO したい相手 + 技（例: ガブリアス をじしんで確 1）
  - 耐えたい攻撃（例: メガルカリオ のインファイト）
出力:
  - 必要最小 SP 配分
  - 残り SP の推奨配分
```

#### F. 素早さ比較統合

ダメージ計算結果に「どちらが先に動くか」を自動表示:
- おいかぜ / トリックルーム 対応
- こだわりスカーフ 対応
- すいすい / すなかき 等の素早さ倍化特性
- ブーストエナジー / こだいかっせい / クォークチャージ

#### G. チームインポート/エクスポート

- Showdown 形式のテキスト貼り付け → 自動パース
- Champions 固有フォーマットへの変換
- URL でのチーム共有

#### H. ワンタップ積み技反映

- つるぎのまい を選択 → 攻撃 +2 を自動反映
- ちょうのまい → 特攻+1、特防+1、素早さ+1 自動反映
- りゅうのまい → 攻撃+1、素早さ+1 自動反映

#### I. 残留ダメージ込み KO 計算

KO 確率にエンドターンダメージを含める:
- ステルスロック / まきびし
- 天候ダメージ（砂嵐）
- やどりぎのタネ / しおづけ
- いのちのたま反動
- たべのこし回復
- ゴツゴツメット接触ダメージ

#### J. 不明スプレッド対応

対戦中に相手の SP 配分が不明な場合、「全可能スプレッドでのダメージ幅」を表示:
- 最小ダメージ（フル耐久振り想定）
- 最大ダメージ（無振り想定）
- 代表的配分でのダメージ

### 5.3 UX 機能

#### K. モバイルファースト設計

**最重要 UX 要件。** 既存ツール最大の弱点。

- レスポンシブ: 縦持ちスマートフォンで完結する UI
- タブ切替: 攻撃側/防御側をタブで切替（2 パネル並列ではなく）

- 大きなタッチターゲット: ドロップダウンではなくモーダル内検索

#### L. ダークモード

- デフォルトでシステム設定に追従
- 手動切替可能

#### M. 計算結果のコピー/共有

- ワンタップで計算結果テキストをクリップボードにコピー
- Discord/X 向けのフォーマット済みテキスト
- URL 共有（計算パラメータをクエリパラメータにエンコード）

#### N. 多言語対応

- 日本語 / 英語 の 2 言語を最低サポート
- ポケモン名・技名・特性名は両言語で検索可能

#### O. オフライン動作

## 6. 既存シミュレーターの分析と差別化

### 6.1 主要ツール詳細比較

#### Smogon Damage Calculator (calc.pokemonshowdown.com)

**強み:**
- 業界標準の精度（オープンソース `@smogon/calc`）
- Gen 1-9 全世代対応
- 大規模な Smogon 推奨セットデータベース
- One vs All / All vs One モード

**弱み:**
- Champions 未対応（SP システム非対応）
- モバイル UX が壊滅的（2 パネル固定レイアウト）
- ダブルバトル対応が不十分
- 天候設定が勝手にリセットされる
- ダークモードなし
- Berry 消費が KO 計算に含まれない
- 残留ダメージが KO 計算に含まれない

#### Pikalytics (pikalytics.com/calc)

**強み:**
- 使用率データ連携（リアルメタ反映）
- 耐久 EV 最適化機能（Survival EV Minimizer）
- セット共有 URL

**弱み:**
- `@smogon/calc` のフォークなので同じバグを継承
- モバイルアプリ有料（$0.99）なのに計算機能なし

#### VGC Multicalc (vgcmulticalc.com)

**強み:**
- 複数ポケモン同時計算（唯一の機能）
- Angular 製のモダン UI
- 完全クライアントサイド

**弱み:**
- VGC 専用（シングル非対応）
- Smogon calc ライブラリ依存
- Champions 未対応

#### Champions Lab (championslab.xyz)

**強み:**
- Champions 対応済み
- AI バトルシミュレーション（200 万戦以上）
- メタ分析・ティアリスト

**弱み:**
- ダメージ計算の詳細度が不明（シミュレーション重視で個別計算が弱い可能性）

## 8. 技術設計方針

### 8.1 アーキテクチャ

```
┌─────────────────────────────────────────────┐
│                    UI Layer                  │
│         (React / Next.js + Tailwind)        │
│    Mobile-first, PWA, Dark Mode, i18n       │
├─────────────────────────────────────────────┤
│               Calculation Engine             │
│         Pure TypeScript, no side effects     │
│    calculate(), calcStat(), optimizeSP()    │
├─────────────────────────────────────────────┤
│                 Data Layer                   │
│         JSON files, version-tagged           │
│    Species, Moves, Items, Abilities, Types  │
└─────────────────────────────────────────────┘
```

### 8.2 技術スタック候補

| 層 | 技術 | 理由 |
|----|------|------|
| 計算エンジン | **TypeScript**（Pure） | 型安全性、テスト容易性、既存エコシステムとの互換 |
| UI フレームワーク | **Next.js (App Router)** or **Remix** | SSG でパフォーマンス、PWA 対応 |
| スタイリング | **Tailwind CSS** | モバイルファースト、ダークモード容易 |
| 国際化 | **next-intl** or **i18next** | 日英対応 |
| テスト | **Vitest** | 高速、TypeScript ネイティブ |
| データ管理 | **JSON ファイル（バージョン管理）** | Git でパッチ追跡、CI/CD で自動更新 |
| デプロイ | **Vercel** or **Cloudflare Pages** | エッジ配信、無料枠あり |

### 8.3 計算エンジン設計原則

1. **純粋関数**: 副作用なし、同じ入力に対して常に同じ出力
2. **不変性**: 入力オブジェクトは clone してから計算
3. **4096 ベース整数演算**: ゲーム内部の端数処理を完全再現
4. **テスト駆動**: 既知の計算結果（Smogon calc との比較）で検証
5. **拡張性**: 新ギミック追加時にコア計算式を変更しない Handler パターン

### 8.4 データ更新フロー

```
バランスパッチ発表
  ↓
data/*.json を更新（手動 or Serebii スクレイピング）
  ↓
CI でテスト実行（既存計算結果が変わらないことを検証）
  ↓
自動デプロイ
  ↓
ユーザーは次回アクセス時に自動反映（PWA の Service Worker 更新）
```
